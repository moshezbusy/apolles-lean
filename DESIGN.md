# Design System

## Product Character

This app should feel operational, premium but restrained, clear, trustworthy, and modern but not trendy.

It is a B2B tool for travel agents managing hotel search, booking, reservations, and booking details. The interface should support fast scanning, low ambiguity, and confidence in operational decisions.

Inside app pages, avoid flashy or marketing-style design. The UI should feel like a polished operations product, not a campaign landing page.

## Visual Principles

1. Clarity first: primary actions and critical data should be immediately understandable.
2. Restraint over decoration: visual quality should come from alignment, spacing, hierarchy, and consistency.
3. Premium through discipline: use clean spacing, stable structure, and measured emphasis instead of trendy effects.
4. Operational scanability: pricing, dates, cancellation terms, statuses, and references must be easy to find.
5. Workflow continuity: related pages should feel like steps within one system.

## Layout

1. Inner app pages should use structured, dashboard-like layouts with clear sections.
2. Use centered content containers and consistent horizontal padding.
3. Prefer stable page frames: top navigation, page header, content sections, side summaries where appropriate.
4. Use cards and bordered panels to group information, not to add decoration.
5. Keep dense information aligned and scannable.

## Spacing

1. Maintain a consistent spacing rhythm between page header, sections, cards, and internal rows.
2. Avoid both cramped layouts and oversized whitespace.
3. Section spacing should feel deliberate and even across workflow pages.
4. Reuse existing vertical spacing patterns before introducing new spacing values.

## Typography

1. Use a restrained heading hierarchy.
2. Headings should support structure, not dominate the screen.
3. Body copy should stay concise and functional.
4. Secondary text should clarify context without competing with primary content.
5. Use typography to improve scanability for labels, totals, dates, and references.

## Buttons and Actions

1. Primary actions should be obvious but not loud.
2. Secondary actions should remain visually subordinate.
3. Reuse the same button hierarchy across pages.
4. Keep action placement consistent for similar workflows.
5. Avoid inventing page-specific CTA styling.

## Forms

1. Inputs, selects, and textareas must follow one consistent treatment.
2. Labels should be clear, compact, and aligned with the existing app style.
3. Required fields, helper text, and validation states should feel systematic.
4. Form layouts should favor completion speed and readability.
5. Avoid decorative input treatments.

## Cards and Summary Panels

1. Use one shared card language for inner app pages.
2. Cards should communicate grouping, not personality.
3. Summary sidebars should highlight key booking and pricing data with clean hierarchy.
4. Reuse existing bordered, rounded, neutral-background containers before creating alternatives.

## Tables and Data Lists

1. Tables and list rows must prioritize alignment and scanability.
2. Dates, currency, status, supplier, and booking references should be visually easy to compare.
3. Keep row density consistent with current reservations patterns.
4. Actions within tables should be easy to identify without clutter.

## Status Badges

1. Status badges should use a single shared visual system across reservations and booking flows.
2. Semantic color should be restrained and readable.
3. Labels should be short and operational.
4. Do not introduce alternate badge shapes, color logic, or casing without a strong system reason.

## Critical Data Clarity

These items must always be visually clear and easy to scan:

1. Booking data
2. Pricing and totals
3. Cancellation policy and fees
4. Stay dates and duration
5. Reservation or confirmation references
6. Guest information
7. Booking status and supplier status

If there is a tradeoff between visual novelty and data clarity, choose clarity.

## Reuse Rules

1. Start with existing pages and shared components.
2. Reuse the closest current pattern before making a new one.
3. If a new pattern is required, align it with the established card, form, badge, table, and spacing system.
4. New patterns must be reusable across future operational pages.
5. One-off visual experiments are not allowed unless explicitly requested.
